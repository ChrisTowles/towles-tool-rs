Claude Code info-gathering layer

Consolidate the version-volatile Claude Code JSONL knowledge into one crate, so Agentboard can watch transcripts in real time — fused with PID / running-process info — and tt-graph can keep doing batch historical analysis, both reading from the same source of truth.

Direct answer to “should it be its own thing?”

Two halves, two answers. The live engine (“what’s actually running”) already exists as tt-agentboard — claude agents --all --json for liveness + procenv /proc PID→session + fs_notify for real-time + JSONL tail for enrichment. Don’t build a second one. What’s missing is a shared schema/parse layer: tt-graph and tt-agentboard currently hand-parse the same transcript lines with two different type sets. Extract that into its own small crate that both consume, and expose it as a CLI/MCP surface you “then use.”

Decision 0 — read first  Do we even do this now?

The honest framing before any of the below: agentboard already has the better real-time engine (demo-verified). This was never a “reuse tt-graph” problem — tt-graph is batch/historical. What’s left is optional schema-dedup cleanup, and its own success gates are “totals identical / snapshots unchanged” — i.e. a migration across two working, heavily-tested crates whose payoff is zero behavior change. So the real first decision is whether to spend the risk now.

  • 0a — Defer: document the two parsers as deliberately parallel, revisit on a trigger. honest default
    No forcing function exists yet (nothing’s broken; the two types are intentionally different projections). Add a one-line cross-reference comment in each, and extract only when a third consumer appears or an actual upstream schema change breaks both at once. Lowest risk; keeps two demo-verified pipelines untouched.
  • 0b — Extract now (the plan below). Worth it only if you expect the schema to churn soon, want the single quarantine point on principle, or have a third consumer coming. Accept the regression risk on a working live pipeline for a cleanup with no user-visible payoff.
  • 0c — Minimal: share only the raw line schema (Decision B3), leave all projections where they are. Middle path — dedups the field definitions without touching agentboard’s hot-path content handling.

Your call, and it gates everything below. If 0a, we stop here with a two-line doc note. Decisions A–D only matter under 0b/0c.

One thing to check that changes the math: does tt-agentboard::claude_usage actually have the #18 double-count bug? It computes a context countdown, not historical totals — if it never sums across the whole transcript, the concrete dedup win doesn’t apply to it, shrinking 0b’s benefit further.

Decisions you’ll want to weigh in on

Decision A  Crate boundary — what goes in the shared crate

Everything hinges on this. It sets what tt-graph is forced to pull in as deps.

  • A1 — Schema + parse + shared projections only. recommended
    tt-claude-code owns the canonical tolerant line types, the JSONL parse, and the pure projections both callers need (title, deduped usage totals). Minimal deps (serde, serde_json, chrono). Live gathering (CLI / proc / notify / tail engine) stays in tt-agentboard.
  • A2 — Whole Claude-Code integration. tt-claude-code also owns the claude --json wrapper, procenv, fs_notify, and a live-snapshot API. Cleaner “one place for CC” story, but tt-graph would drag in notify/proc deps it never uses.
  • A3 — No new crate; agentboard depends on tt-graph. Rejected: wrong direction — the real-time crate would inherit tt-graph’s regex + HTML-render + treemap deps, and the dependency arrow points backwards (batch analytics shouldn’t be upstream of the live engine).

Your call: A1 keeps tt-graph lean; pick A2 if you’d rather have a single “all things Claude Code” crate even at the cost of heavier deps.

Decision B  The canonical type + how much logic moves down

Today the same line is modelled twice. Collapsing them is the core data-model change — here’s the shape and the one real friction point.

// tt-claude-code: the ONE tolerant transcript line (replaces tt-graph::JournalEntry
// AND tt-agentboard::watchers::claude_code::RawEntry). All fields optional.
pub struct TranscriptEntry {
    pub entry_type: String,           // "type"
    pub timestamp: Option<String>,
    pub session_id: Option<String>,   // "sessionId"
    pub request_id: Option<String>,   // "requestId"  ← dedup key half (#18)
    pub git_branch: Option<String>,   // "gitBranch"
    pub slug: Option<String>,
    pub custom_title: Option<String>, // "customTitle" (#17)
    pub ai_title: Option<String>,     // "aiTitle"     (#17)
    pub message: Option<Message>,     // { id, role, model, usage, content }
}

// Pure projections both callers import (the volatile knowledge, in one place):
pub fn parse_transcript(content: &str) -> Vec<TranscriptEntry>;
pub fn session_title(entries: &[TranscriptEntry]) -> Option<String>;
pub fn usage_totals(entries: &[TranscriptEntry]) -> UsageTotals; // deduped by message.id:requestId
pub fn dedup_key(entry: &TranscriptEntry) -> Option<String>;

The friction: tt-graph keeps content as an untyped Value array; tt-agentboard parses content blocks into typed items (tool_use name/input) to derive status, last-tool, thread-name, /loop wakeups. So “how much moves down” is a genuine choice:

  • B1 — Shared schema keeps content untyped; shared crate also ships typed accessor helpers. recommended
    tt-claude-code exposes entry.text_blocks(), entry.tool_uses() etc. Agentboard’s status/thread/loop logic is rebuilt on those helpers — so there’s exactly one parse of content, but the agent-semantics (what “busy/waiting” means) stay in agentboard where they belong.
  • B2 — Shared crate owns the semantics too (derive_status, thread_name, context_countdown). Maximum dedup, but now “what counts as waiting” is upstream of the UI that defines it — likely to churn.
  • B3 — Share only the raw schema; each crate keeps its own content handling. Least disruption, but the content-block shape (also volatile) stays duplicated — half-solves the problem we’re here to solve.

Your call: B1 is the balance point (one parse, semantics stay local). Say the word if you’d rather pull status logic all the way down (B2) or keep the blast radius tiny (B3).

Decision C  Real-time model & “what’s actually running” (user-facing)

This is what you’ll actually see in Agentboard, so it’s worth agreeing on the semantics.

Liveness = CLI + PID, not JSONL

Source of truth for “running” is claude agents --json ∪ /proc claude procs carrying TT_SESSION_ID. JSONL never decides liveness — only enriches (model, tokens, last tool, title).

Real-time = notify + poll fallback

fs_notify triggers an eager rescan on transcript change; the periodic scan is the reliable floor. Tail reads resume from the last newline offset (cheap on huge files).

The semantic choice — a session whose JSONL just changed but that the CLI/PID scan does not list:

  • C1 — Not running. recommended Honours “what’s actually running”; a recently-touched transcript with no live process is shown as idle/ended, not active.
  • C2 — “Recently active” grace window. Treat mtime within N seconds as still-live to smooth races. Softer, but can show ghosts after a crash.

Tweakable knobs (keep current values unless you want to change them): watcher tick 2s · CLI snapshot cache 1.5s · pane scan 3s · notify debounce.

Your call: C1 vs C2, and whether any of those cadences should change.

Decision D  Standalone surface — the “that I then use” part

You asked for a thing you can use, not just an internal lib. Options for exposing the layer:

  • D1 — Add ttr cc subcommands over the live API. recommended
    ttr cc sessions [--json] → live snapshot (pid, cwd, session, status, model, context-left, title); ttr cc watch → stream changes. Makes the layer scriptable and testable outside the app.
  • D2 — Library + MCP only. tt-mcp already exposes live sessions; point it at the new layer and stop there. Less surface to maintain; nothing on the CLI.
  • D3 — Both. CLI now, and re-point MCP + the Tauri store_snapshot path at the same API so all three read one source.

Your call: is a ttr cc CLI worth it, or is MCP/app consumption enough?
