Agentboard — lifecycle, cache health & windows

Implementation plan. Decisions you'll want to tweak lead; mechanical work is at the bottom.

Guiding realisation. The app already routes every keystroke into the PTY via term_write, and CLAUDE.md's own rule is "agent status is reported, never re-rendered — interaction happens in the real PTY." So start / stop / compact / restart are mostly PTY writes plus an optimistic UI overlay, not new process-control commands. That's why the "ambitious" lifecycle work turns out cheap. The genuinely new backend is small: extend the emitted payload with cache/context fields (already parsed) + a live flag, and add a folder-purpose setter. Everything else is frontend.

Part 1 · Decisions to review The choices with product judgment in them

1 · Start / Stop / Restart / Compact — what they actually do your call

Do these drive the PTY, or signal the process by PID?
Recommend — PTY writes. start = term_write(id,"claude\r") into the session's shell · stop = term_write(id,"\x03") (Ctrl-C → SIGINT to the foreground process, shell survives) · compact = term_write(id,"/compact\r") · restart = Ctrl-C then claude --continue\r. No new backend command, and it behaves exactly like you typing it.

Alternative — PID signals. ab_stop_agent looks up the claude pid (we already map session→pid in procenv.rs) and sends SIGINT. More "correct" for a stuck TUI, but Linux-only, and it fights the PTY-is-truth model. I'd keep this as a fallback only if Ctrl-C proves unreliable.

Open sub-questions for you: (a) does start need the shell already live, or should it auto-term_start a stopped session first? (recommend: auto-start) · (b) compact only makes sense at a Claude prompt — do we gate it on status==="waiting"|"idle", or fire-and-hope? (recommend: gate) · (c) is restart = --continue (resume thread) or a truly fresh session? (recommend: fresh; add --continue later as a second button if wanted).

2 · Cache health & the compact nudge your call

When is a session "cold", and when do we nag?
Recommend. cold = !cacheExpiresAt || now ≥ cacheExpiresAt. Show the ❄ compact nudge only when cold AND ctx% ≥ compactRecommendPercent. Honour the real TTL: a 5-minute cache shows ◔ m:ss, a 1-hour ephemeral cache shows a calmer ⧗ — both are in the data.

Decision: what does cacheExpiresAt == null (a turn with no cache activity) mean — cold (recommend, it's the safe read) or unknown/hidden? · Should the nudge also fire while warm if ctx is very high (say ≥80%)? I'd say no — warm-and-huge is fine to keep going; the cost only bites on a cold resume.

3 · The threshold is a setting, exposed in Agentboard settings your call

Where does compactRecommendPercent live and default?
Recommend. New agentboard.compactRecommendPercent in tt-config (default 30), surfaced by a slider in an Agentboard settings panel reachable from the window strip's ⚙ chip. Because the settings file is shared with the TypeScript CLI, the field is added with #[serde(default)] so the other tool never breaks on it.

Decision: global, or per-repo override? Recommend global-only for v1.

4 · Session lifecycle states in the rail your call

running · not-started · no-sessions — and do we still auto-seed a shell?
Recommend. A session shows a filled dot when its PTY is live and a hollow ring when not started. Allow a folder to have zero sessions (the "— no sessions —" empty state). Keep the default-shell 1 seed, but it starts not-started (a record, no PTY) instead of implying activity.

Decision: should adding a repo auto-seed shell 1 at all, or land folders truly empty and let you open the first session? Recommend keep the seed but not-started.

5 · The rolled-up tally on the nav your call

What does the Agentboard nav entry summarise?
Recommend. N agents + per-status mini-dots for the non-zero buckets (●busy ●waiting ●error), plus a distinct ❄ K to compact when any session crosses the threshold. Same math powers the rail header chip.

Decision: count only running agents (recommend) or also idle shells? And does the compact count belong on the nav, or only inside the board? Recommend: on the nav — it's the "come back to me" signal.

6 · Windows, panes & rail grouping your call

All in-app (settled). How much of it persists, and how?
Recommend. Windows are frontend-owned layout state (a window = a named, ordered list of pane session-ids), tiled side-by-side, persisted through one debounced setter ab_save_windows(json) and returned in ab_get_state. The rail shows each open session's window colour tag; the tag ↔ window-strip colours match; hover-⊟ ungroups (removes the pane, session stays in the rail).

Decision: a chatty command surface (ab_window_create/close/rename/add_pane/remove_pane) vs the single ab_save_windows blob (recommend the blob — layout changes are frequent and low-stakes) · Decision: when a grouped session is deleted, its pane just vanishes — fine? (recommend yes) · Decision: default tiling for 3 panes — thirds across, or 2-over-1? (recommend thirds; 2×2 at four).

type AgentUsage = {
  contextUsed: number; contextMax: number;
  cacheExpiresAt?: number | null; cacheTtlMs?: number | null; lastActivityAt: number;
};
type AgentEvent  = { /* …existing… */ usage?: AgentUsage | null };
type SessionData = { /* …existing… */ live: boolean };
type FolderData  = { /* …existing… */ purpose?: string | null };

type AgWindow   = { id: string; name: string; panes: string[] };  // panes = session ids
type WindowsPayload = { windows: AgWindow[]; activeWindow: string };

// derived selectors (pure, colocated):
const ctxPct      = (u,cap=200000) => u ? Math.round(u.contextUsed / u.contextMax * 100) : 0;
const isCold      = (u,now) => !u?.cacheExpiresAt || now >= u.cacheExpiresAt;
const needsCompact= (u,now,pct) => !!u && ctxPct(u) >= pct && isCold(u,now);

Part 5 · Mechanical work Trusted to me — the plumbing

  • Serde rename_all = "camelCase" on all new payload structs; verify the TS mirror matches field-for-field.
  • Persist-file IO in tt-agentboard: atomic write (temp + rename), tolerate-missing on read, no unwrap on IO (crate rule).
  • Debounce ab_save_windows on the frontend (~400ms) so drag/open/close don't thrash the disk.
  • Thread the optimistic-overlay clock off a single now tick already used for cache countdowns.
  • Unit tests: needsCompact boundary (exactly at threshold, null cache, 1h vs 5m TTL); liveness merge; purpose round-trip; windows blob round-trip.
  • cargo fmt --check · cargo clippy --all -D warnings · cargo test --all; commit messages cite slot-1 / yaak source where derived.
  • Keep every hue driven by statusColor() + the amber/violet/ice accents; add shadcn primitives via the CLI, no hand-rolled CSS (folder-rail rule).
